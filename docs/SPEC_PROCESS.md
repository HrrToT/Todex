# Todex 规约过程（SPEC_PROCESS）

状态：in progress
最后更新：2026-07-13

## 目的

本文件记录使用 Superpowers 从初始方向到规约和计划的过程。它将保存关键追问、项目负责人的决定、被推翻的方案、冷启动验证暴露的歧义以及由此产生的 SPEC/PLAN 修订。

## 已记录的关键迭代

### 说明：对话节选的记录方式

以下节选基于本次 brainstorming 的真实对话整理，保留问题、选择和判断链条，不伪造逐字转录。`“项目负责人”`指本项目学生；`“主导 agent”`指使用 Superpowers 的 Codex。

### 2026-07-13：从课程方向到产品边界

初始方向是“Todex：小型代码仓库的轻量 coding agent harness”，能力包括读写文件、执行测试、接收反馈和自我修正。

决定：选择真实开发者导向而非单纯机制演示；产品以本地 WebUI 为主，并最终交付 Windows Electron 安装包。

**对话节选与处理：**

- 主导 agent 追问：产品是机制演示，还是让真实开发者在小仓库中修 bug、改代码？
- 项目负责人选择：真实开发者导向。
- 产生的规约修订：SPEC 不再把 Mock LLM 演示站当成全部产品，而明确区分本地真实 Electron 产品与线上受限 Demo。

### 2026-07-13：项目探测、模型和治理

决定：不要求用户预先声明仓库类型。Todex 自动探测项目，再向用户展示候选测试命令并要求确认。

决定：只实现一个 OpenAI-compatible LLMClient，支持 GLM、DeepSeek、Qwen 等兼容供应商；API Key 使用 Windows Credential Manager。

决定：采用类似 Codex 的治理方式，安全操作默认允许，高风险操作必须经过一次、本轮或同命令前缀粒度的审批，并记录在 trace 中。

**对话节选与处理：**

- 项目负责人质疑：为什么用户必须显式声明项目类型，日常 coding agent 通常不需要？
- 主导 agent 提出：自动探测后交互确认命令，避免用户配置负担与误执行风险。
- 项目负责人选择：自动探测 + 交互确认。
- 产生的规约修订：项目类型只作为探测结果，测试/质量命令必须经确认后成为 `ConfiguredCommand`；模型不能任意拼接命令。

### 2026-07-13：范围、测试与分发

决定：V1 对 Node.js 与 Python 都提供完整的探测和反馈闭环支持。

决定：核心机制的自动化验收使用确定性的 Mock LLM，不依赖网络或真实模型；真实模型是桌面产品功能，不是课程测试前提。

决定：采用共享 `harness-core` 的 TypeScript monorepo。Windows Electron 是真实本地产品宿主；公网 WebUI 是不接收真实 Key、仅运行内置示例仓库和 Mock LLM 的演示宿主。

**对话节选与处理：**

- 主导 agent 曾将“真实模型现场演示”作为可选证据提出。
- 项目负责人纠正：课程没有现场演示环节。
- 主导 agent 重新阅读原始要求后确认：课程需要的是确定性的 mock/stub 测试、机制演示和可访问 WebUI，而不是依赖一次真实模型调用。
- 产生的规约修订：自动化验收只依赖 Mock LLM；真实模型仅作为桌面端功能。公网 WebUI 使用内置示例和 Mock LLM，避免云端访问用户本地仓库或存储真实 Key。

### 2026-07-13：版本边界与开发阶段

决定：本课程只承诺 V1.0。V1.0 先完成规约冷启动、内核、治理与反馈、Node/Python 适配、桌面端与凭据、Demo/打包/CI 的阶段性闭环。Git worktree、diff 回滚、运行时多模型协作、团队能力、跨平台和远程执行器明确推迟到 V1.1、V1.2 或 V2.0，避免未来愿景侵蚀课程交付范围。

### 2026-07-13：治理与 HITL 机制确认

决定：治理采用不可绕过的动作路径、工作区真实路径围栏、敏感文件硬拒绝、允许/审批/拒绝三级风险分类和可审计 HITL 状态机。项目级同前缀许可采用保守的规范化命令指纹，默认 7 天失效；网络、删除、安装和破坏性 Git 操作不支持项目级许可。该决定将作为 V1.0 的主要深入机制，以 Mock LLM、Fake Clock 和 Fake Command Runner 做确定性测试。

### 2026-07-13：反馈、记忆与数据模型确认

决定：测试、lint、typecheck 与 build 由固定 `commandId` 驱动，作为确定性反馈传感器；修复上限为 3，环境问题不被误当作代码 bug。轻量记忆不采用 RAG，只保存可解释的项目事实与带 trace 证据的 Agent 观察。桌面端使用应用数据目录 SQLite，真实 API Key 只存 Windows Credential Manager。

### 2026-07-13：用户故事与 WebUI 确认

决定：V1.0 覆盖工作区探测、任务 trace/diff、审批、自修复、模型凭据、项目记忆和公网 Mock Demo 七条独立用户故事。WebUI 采用开发工具工作台布局，桌面端与 Demo 共享结构但不共享权限；前端设计方法选择 Open Design，并在实际前端开发前记录 skill 的安装和使用。

### 2026-07-13：安全、质量与验收确认

决定：Todex 定义为工作区级治理工具而非 OS 级沙箱；威胁模型覆盖 Key、敏感文件、路径逃逸、prompt injection、命令绕过、供应链和 Demo 滥用。V1.0 明确支持 Windows 10/11 x64，安装包默认未签名并在 README 透明说明 SmartScreen 风险。核心验收以 Mock LLM、示例仓库、Electron 安装包、GitHub Actions 和公网 Mock WebUI 的客观行为为准。

### 2026-07-13：最终规约签字

项目负责人确认 Todex V1.0 的完整设计，包括范围、架构、技术选型、治理、反馈、记忆、UI、安全、分发、验收与路线图。下一阶段切换到 `writing-plans`，随后执行陌生 agent 冷启动验证；在此之前不得开始实现代码。

## AI 建议、人工判断与阶段性反思

### 被采纳的建议

- 共享 `harness-core` 的 monorepo：让 Electron 与公网 Demo 使用同一套可单测机制。
- Mock LLM 与真实 LLM 使用同一接口：降低 CI、冷启动与机制演示对网络和额度的依赖。
- 治理作为主要深入维度：比把六类 Harness 能力平均铺开更能满足“机制必须是代码”的要求。
- 项目级轻量记忆：先实现可解释、可删除、按需加载的事实记忆，推迟 RAG。

### 被项目负责人修正的建议

- 不能把真实模型调用或录屏当作课程验收前提，因为课程没有现场演示环节。
- 不要求用户手工声明 Node/Python 类型，改为自动探测与交互确认。
- GitHub 是唯一实际协作与 CI 平台；GitLab 不作为运行平台。

### 阶段性反思

brainstorming 对发现隐性边界很有效，尤其是“线上 WebUI 如何安全访问本地仓库”“项目类型是否要求显式配置”“真实模型是否能作为测试前提”这几处。它的不足是容易把合理的工程建议说得像课程硬性要求；真实阅读原始任务文件并接受项目负责人纠正，才让规约回到课程实际约束。后续冷启动验证将检验这些文字是否足够让陌生 agent 不依赖本对话也能执行。
